# 项目改造开发计划

## 1. 背景

当前仓库同时承载了几类能力：

- `SOP Chat` 主服务：提供 Web UI、配置管理、OpenAI 兼容接口、多 IM 渠道接入、定时任务等运行时能力
- 渠道接入：钉钉、飞书、企业微信、企业微信群机器人
- 配置与认证：统一配置、内置认证、配置管理 UI、热重载
- 辅助能力：`skills/generate-sls-sop` 等离线 SOP 生成技能与示例文档

随着功能持续堆叠，当前代码已经出现明显的边界模糊、文件过大、测试覆盖不足和兼容逻辑持续累积的问题。后续如果继续直接叠加需求，维护成本和改造风险都会快速上升。

本计划用于统一后续改造方向，避免边改边漂移。

## 2. 当前主要问题

### 2.1 模块职责过重

当前存在多个过大的核心文件：

- `backend/internal/config/config.go`
- `backend/internal/api/configui.go`
- `backend/internal/api/server.go`
- `frontend/src/components/ChatWindow.jsx`

这些文件同时承担了模型定义、协议适配、兼容逻辑、业务编排、状态管理等多种职责，导致：

- 修改影响范围难以预估
- 代码复用困难
- 测试边界不清晰
- 新人接手成本高

### 2.2 配置系统耦合过深

当前配置链路横跨多个层面：

- YAML 配置结构
- Go 配置模型
- Config UI DTO
- 前端表单状态
- 热重载逻辑
- 运行时解析逻辑

主要问题：

- 配置项新增或调整时改动面过大
- 持久化模型与展示模型未明显分层
- 向后兼容字段持续累积
- 敏感字段处理边界不清

### 2.3 前后端聊天协议耦合偏紧

聊天流能力目前依赖 SSE 协议，但前端对后端事件结构感知较深，导致：

- 前端状态管理复杂
- 后端事件一旦调整，前端容易跟着出问题
- 工具调用展示逻辑不易复用和测试

### 2.4 业务主链路测试不足

现有测试更偏向 SDK 或局部模块，核心业务能力缺口较大，包括但不限于：

- 配置保存与热重载
- 运行时解析与路由
- 聊天流式协议
- OpenAI 兼容接口
- 前端聊天状态管理

### 2.5 安全与可运维性需要收敛

当前系统中存在较多敏感配置字段和运行日志输出，后续若继续扩展，需要统一治理：

- 配置返回是否脱敏
- 日志中是否可能出现敏感字段
- 测试样例和示例配置是否存在泄漏风险
- 多渠道接入后的排障手段是否统一

### 2.6 仓库边界需要明确

仓库同时包含主服务与 `skills/` 能力，但二者生命周期不同：

- 主服务强调运行时稳定性
- `skills/` 更偏离线文档生成与工程脚本

如果边界不明确，后续会在发布、测试、文档和目录结构上持续互相牵制。

## 3. 改造目标

本轮改造优先解决以下问题：

1. 建立清晰的模块边界
2. 降低配置系统改动成本
3. 固化聊天协议与运行时契约
4. 为后续功能扩展建立统一规范
5. 补齐核心链路测试
6. 收敛安全、兼容和日志策略

## 4. 非目标

以下事项暂不作为本轮改造第一优先级：

- 一次性重写全部前端页面
- 一次性替换所有历史配置格式
- 直接拆分成多仓库
- 立即补齐所有未实现认证模式
- 先做大规模视觉改版

这些内容可以纳入后续阶段，但不应阻塞当前架构梳理和基础收敛。

## 5. 总体改造策略

### 5.1 先拆边界，再改实现

先把配置、运行时、API、渠道、前端状态的边界整理清楚，再逐步迁移代码。避免在大文件中边写新逻辑边做结构调整。

### 5.2 先稳定主链路，再处理外围能力

优先级顺序建议如下：

1. 配置系统
2. 运行时解析与路由
3. Chat / Thread / Employee 主链路
4. SSE 协议与前端聊天状态
5. OpenAI 兼容层
6. 渠道机器人
7. Scheduler
8. `skills/` 目录治理

### 5.3 兼容读优先，收口写路径

对历史字段和旧配置尽量保持读取兼容，但新的保存逻辑、UI 提交逻辑和默认生成逻辑应尽量收口到新结构，避免兼容继续扩散。

## 6. 分阶段计划

### 阶段一：基线梳理

目标：把边界、问题、优先级和约束写清楚。

交付物：

- `docs/项目改造开发计划.md`
- `docs/项目开发规范.md`
- 改造任务拆分草案

完成标志：

- 团队对本轮改造范围、优先级和规范达成一致

### 阶段二：配置系统拆层

目标：将配置相关职责从“大一统文件”中拆出来。

优先动作：

- 区分配置持久化模型、运行时模型、UI 交互模型
- 提取配置校验与默认值逻辑
- 提取兼容字段转换逻辑
- 为敏感字段增加统一脱敏策略

建议目录方向：

- `backend/internal/config/model`
- `backend/internal/config/loader`
- `backend/internal/config/validator`
- `backend/internal/config/migrate`
- `backend/internal/config/mask`

完成标志：

- 新增配置项不再需要跨多个超大文件手工同步修改
- 配置 UI 和持久化模型之间有明确转换层

### 阶段三：运行时与 API 主链路收敛

目标：降低 `server.go`、`configui.go`、`employee.go`、`thread.go` 等文件的编排复杂度。

优先动作：

- 提取 runtime 解析层
- 提取渠道生命周期管理层
- 提取 handler 对应的 service 层
- 统一错误返回与日志格式

建议目录方向：

- `backend/internal/runtime`
- `backend/internal/service`
- `backend/internal/api/handler`
- `backend/internal/api/dto`

完成标志：

- handler 只负责参数解析、调用 service、返回响应
- runtime 解析不再散落于各接口和渠道实现中

### 阶段四：聊天协议与前端状态重构

目标：降低前端对 SSE 事件细节的耦合。

优先动作：

- 抽离聊天事件协议定义
- 前端引入事件适配层
- 将 `ChatWindow.jsx` 重构为状态层 + 展示层
- 为工具调用展示建立独立数据结构

建议目录方向：

- `frontend/src/features/chat`
- `frontend/src/features/chat/protocol`
- `frontend/src/features/chat/state`
- `frontend/src/features/chat/components`

完成标志：

- 前端不直接依赖后端内部事件细节
- 聊天主界面文件明显缩小

### 阶段五：测试补齐与回归保护

目标：给关键改造点补上自动化保护。

优先级：

1. 配置加载、校验、迁移
2. 运行时解析与路由
3. Chat SSE 协议
4. OpenAI 兼容接口
5. 前端聊天状态与协议解析

完成标志：

- 核心链路具备最基本的回归测试
- 新增变更有明确测试归属

### 阶段六：外围能力治理

目标：处理渠道、定时任务、`skills/` 和文档结构的收口问题。

优先动作：

- 统一渠道实现接口
- 收敛 Scheduler 触发链路
- 明确 `skills/` 与主服务边界
- 补齐运维和开发文档

完成标志：

- 仓库结构更容易理解
- 主服务与离线能力的职责边界明确

## 7. 里程碑建议

### 里程碑 M1：完成文档与任务拆分

验收标准：

- 明确第一批改造对象
- 明确不做事项
- 明确规范

### 里程碑 M2：配置系统完成第一轮拆层

验收标准：

- 配置模型与 UI DTO 分层
- 敏感字段策略落地
- 基础测试补齐

### 里程碑 M3：聊天主链路完成收敛

验收标准：

- 后端主链路职责清晰
- SSE 协议收口
- 前端聊天状态重构完成第一轮

### 里程碑 M4：外围模块进入稳定迭代

验收标准：

- 渠道、Scheduler、OpenAI 兼容层具备统一开发模式
- 后续需求可按规范增量演进

## 8. 风险与应对

### 风险一：边重构边加需求导致结构继续恶化

应对：

- 改造期间新增需求优先走新边界
- 对旧文件只做必要修复，不继续叠加复杂逻辑

### 风险二：兼容历史配置时复杂度失控

应对：

- 保持读取兼容
- 收口写路径
- 为兼容逻辑增加注释和淘汰计划

### 风险三：缺少测试导致回归频发

应对：

- 每完成一个关键拆分，就同步补测试
- 修复问题必须补回归测试

### 风险四：改造范围膨胀

应对：

- 按阶段推进
- 每个阶段都要求有明确验收边界
- 不把所有历史问题一次性塞进同一轮改造

## 9. 第一批建议任务

建议优先落地以下任务：

1. 拆分配置模型与 UI DTO
2. 设计运行时解析层
3. 抽离聊天 SSE 协议定义
4. 重构 `ChatWindow.jsx` 状态管理
5. 为配置和聊天主链路补基础测试
6. 梳理敏感字段脱敏与日志规范

## 10. 讨论待定项

以下问题建议在进入实作前进一步确认：

1. `skills/` 是否继续与主服务同仓库维护
2. 是否需要在本轮改造中引入 `service` / `runtime` 分层
3. 前端是否接受按 feature 目录重组
4. 历史配置兼容计划保留到哪个阶段
5. OpenAI 兼容层是否继续作为主服务一部分长期维护

## 11. 结论

当前项目最需要的不是继续堆功能，而是先建立清晰、可演进的边界和规范。

本计划的核心原则是：

- 先收敛，再扩展
- 先稳定主链路，再治理外围
- 先建立规则，再推进大规模改造

后续具体任务拆分、里程碑和提交节奏，都应以本计划为基线持续细化。
