# 项目开发规范

## 1. 目的

本规范用于约束后续改造和新功能开发，目标是提升以下几项能力：

- 可维护性
- 可测试性
- 可审查性
- 可演进性
- 安全性

本规范优先适用于本仓库中的以下范围：

- `backend/`
- `frontend/`
- `skills/`
- `docs/`

## 2. 基本原则

### 2.1 增量改造

不做无边界的大重写。所有改造优先采用增量方式推进，要求每一步都具备：

- 明确目标
- 明确影响面
- 明确回滚方式
- 明确测试归属

### 2.2 单一职责

每个模块、文件、函数、组件都应尽量只有一个主要职责。禁止继续向超大文件叠加横向能力。

### 2.3 读兼容，写收口

历史结构可以兼容读取，但新逻辑必须逐步收口到新结构，禁止边新增功能边扩散旧字段和旧行为。

### 2.4 先定义契约，再写实现

以下内容在开发前应优先定义清楚：

- 配置模型
- API DTO
- SSE 事件协议
- 前端状态结构
- 模块输入输出

## 3. 目录与模块规范

### 3.1 后端分层约束

建议职责划分如下：

- `config`：配置模型、加载、校验、迁移、脱敏
- `runtime`：将配置解析为运行时对象
- `service`：承载业务编排
- `api`：仅负责 HTTP 协议适配
- `channel` 或现有渠道目录：承载各渠道实现
- `scheduler`：定时任务执行与调度

### 3.2 前端分层约束

前端建议按 feature 组织，而不是持续按页面堆积逻辑。

推荐分层：

- `features/*/components`
- `features/*/state`
- `features/*/protocol`
- `services`
- `utils`

### 3.3 `skills/` 目录约束

`skills/` 视为相对独立的交付物，应遵循以下规则：

- 不直接耦合主服务运行时代码
- 变更说明要独立记录
- 脚本、规则、评估数据分层清晰

## 4. 文件规模规范

### 4.1 文件长度控制

建议标准：

- 单文件尽量不超过 400 行
- 超过 600 行必须评估拆分
- 超过 800 行原则上必须拆分，除非是明确的数据定义文件

### 4.2 函数与组件长度控制

建议标准：

- 单函数尽量不超过 80 行
- 单 React 组件尽量不超过 300 行
- 复杂状态逻辑不直接堆在页面组件中

## 5. 配置规范

### 5.1 配置模型分离

至少区分以下三类模型：

- 持久化配置模型
- 运行时配置模型
- 前端展示与提交模型

禁止将三者长期混用。

### 5.2 默认值与校验

要求：

- 默认值集中管理
- 校验逻辑集中管理
- 错误提示可定位到具体字段
- 不允许把校验逻辑散落在多个 handler 和组件中

### 5.3 兼容字段管理

所有兼容字段必须记录：

- 来源版本或来源背景
- 当前兼容原因
- 新结构映射关系
- 计划移除时机

如果做不到这四点，不应继续新增兼容字段。

### 5.4 敏感字段处理

以下信息默认视为敏感信息：

- AccessKey / Secret
- Token
- API Key
- 密码
- Webhook Secret
- Bot Secret

要求：

- 默认不在日志中明文输出
- 默认不在列表接口中明文返回
- 编辑场景如需展示，必须脱敏
- 示例配置统一使用占位符

## 6. API 与协议规范

### 6.1 Handler 约束

Handler 只做以下事情：

1. 参数解析
2. 鉴权与权限检查
3. 调用 service
4. 错误映射
5. 响应输出

禁止在 handler 中直接承载复杂业务编排、配置拼装和兼容转换。

### 6.2 DTO 约束

API 输入输出结构应独立定义，不应直接暴露内部模型。

要求：

- 请求 DTO 与响应 DTO 明确
- DTO 命名统一
- 兼容字段只存在于 DTO 或转换层，不污染内部核心模型

### 6.3 SSE 协议规范

聊天流式协议必须有单独的协议定义，至少明确：

- 事件类型
- 字段结构
- 生命周期顺序
- 错误事件格式
- 兼容策略

前端不得直接依赖“隐式字段”或“后端内部消息细节”。

### 6.4 错误返回规范

错误返回应保持统一结构，至少包括：

- 错误码
- 错误消息
- 可选细节
- 请求上下文标识

禁止同一类错误在不同接口中使用完全不同的结构。

## 7. 前端开发规范

### 7.1 状态管理

以下情况必须考虑独立状态层或 `useReducer`：

- 流式消息
- 工具调用事件
- 多阶段加载状态
- 撤销、重试、终止等交互

### 7.2 组件职责

组件分工建议：

- 页面容器：组织页面结构和数据注入
- 展示组件：只负责渲染
- 状态模块：负责状态转移
- 协议适配层：负责后端数据转前端结构

### 7.3 调试代码

要求：

- 调试逻辑不得污染正式业务流程
- 开发期调试输出应受环境变量或开关控制
- 合并前清理无意义的临时日志和注释

### 7.4 UI 文案与图标规范

正式 UI 必须保持专业、克制、统一，禁止使用表情符号充当视觉设计或功能标识。

要求：

- 禁止在导航、按钮、标题、卡片标题、Tab、表单标签、空状态、提示语、菜单项中使用 emoji
- 明确禁止出现类似 `👥`、`✨`、`🚀`、`📚`、`⚙️` 这类装饰性表情作为正式 UI 文案的一部分
- 如需表达功能含义，优先使用统一的 SVG 图标或图标组件，不允许用 emoji 代替图标系统
- 用户输入内容、业务数据原文、聊天消息正文中的 emoji 不受此限制，但系统界面文案默认禁止使用
- 文案必须简洁、直接、可理解，避免“点击这里试试看”“神奇体验”“一键搞定”这类营销化或口语化表达

### 7.5 UI 视觉设计规范

前端界面必须朝现代化、耐看、专业、易操作的方向设计，不接受“能用就行”的半成品界面。

要求：

- 新页面或大改版必须具备明确的视觉层级，包括标题层级、信息分组、主次按钮、间距节奏
- 优先建立统一设计变量，例如颜色、字号、圆角、阴影、边框、间距、状态色，避免组件各自为政
- 视觉风格应保持现代、克制、有识别度，不允许出现明显拼凑感、默认浏览器风格或过时后台风样式
- 保持对齐、留白、密度一致，避免按钮大小不统一、卡片边距杂乱、输入框和标题不在同一视觉体系
- 颜色使用应有主色、辅色、状态色分工，不允许随意添加高饱和颜色或把多种强调色混用
- 如果项目内已存在设计语言，应优先延续；若现有设计质量较低，可在保持兼容的前提下逐步升级

### 7.6 易用性与交互规范

界面不只要“好看”，还必须“顺手”。所有交互设计都应优先降低理解成本和操作成本。

要求：

- 用户高频操作必须显眼、路径短、反馈明确，不应藏在二级入口或依赖猜测
- 表单页面应提供清晰的字段说明、默认值、错误提示和提交反馈
- 异步操作必须有加载态、成功态、失败态，禁止让用户在无反馈状态下等待
- 空状态必须说明当前为什么没有数据，以及下一步可以做什么
- 危险操作必须有确认机制，普通操作不应过度打断用户流程
- 交互反馈要及时，包括 hover、focus、disabled、error、success 等状态

### 7.7 响应式与可访问性规范

要求：

- 页面和核心组件必须同时适配桌面端与移动端，不能只在开发机宽屏下可用
- 不允许出现移动端溢出、遮挡、按钮难以点击、弹层超出视口等基础问题
- 表单、按钮、链接、弹窗必须具备清晰的焦点状态
- 文本与背景必须具备足够对比度，避免“看起来高级但读不清”的配色
- 图标按钮、纯视觉控件应具备可理解的名称或提示，避免只靠颜色或位置传达含义

### 7.8 评审标准

前端改动在合并前，除功能正确外，还应额外检查以下项目：

- 是否出现 emoji 风格的系统界面文案
- 是否具备统一、现代的视觉层级
- 是否在桌面端和移动端都可正常使用
- 是否存在明显难用、难找、难理解的交互
- 是否保留了调试残留、临时样式或半成品占位设计

## 8. 后端开发规范

### 8.1 日志规范

日志应满足：

- 可定位
- 可筛选
- 不泄露敏感信息
- 不重复刷屏

建议统一记录：

- 模块名
- 操作名
- 关键标识
- 错误摘要

### 8.2 错误处理

禁止：

- 空 `catch`
- 吞错不处理
- 只打印日志不返回错误
- 返回模糊错误导致无法定位

要求：

- 业务错误与系统错误区分处理
- 保留必要上下文
- 错误包装不要丢失根因

### 8.3 Service 层

Service 层负责：

- 业务编排
- 运行时对象组合
- 领域规则执行
- 调用外部依赖

禁止在 service 中混入 HTTP 细节和前端兼容逻辑。

## 9. 测试规范

### 9.1 测试优先级

优先覆盖以下高风险区域：

1. 配置加载、校验、迁移
2. runtime 解析与路由
3. 聊天 SSE 协议
4. OpenAI 兼容接口
5. 定时任务触发链路
6. 前端聊天状态与协议适配

### 9.2 回归要求

要求：

- 修复 bug 必须补回归测试
- 新增协议字段必须补契约测试
- 配置兼容逻辑必须配 fixture

### 9.3 测试命名

要求：

- 测试名称直接反映行为
- 一条测试只验证一类核心断言
- 避免把多个不相关场景塞进同一个测试

## 10. 文档规范

### 10.1 文档分类

建议至少区分：

- 架构文档
- 改造计划
- 开发规范
- 运维文档
- 用户文档

### 10.2 文档命名

本仓库中的 Markdown 文档默认使用中文命名，除非有以下例外：

- 行业标准文件名必须固定，例如 `README.md`
- 第三方集成要求固定命名
- 工具链明确依赖英文文件名

### 10.3 文档更新原则

以下变更必须同步更新文档：

- 配置结构调整
- API 协议调整
- 目录结构调整
- 新增模块边界
- 兼容策略变化

## 11. 提交与评审规范

### 11.1 提交粒度

要求：

- 一个提交只解决一类问题
- 结构调整与功能改动尽量分开提交
- 大重构按阶段拆分提交

### 11.2 提交信息

提交信息遵循仓库既有约定：

- 格式：`<type>(<scope>): <subject>`
- 提交信息使用简体中文

示例：

- `refactor(core): 拆分配置模型与校验逻辑`
- `test(api): 补充聊天流接口回归测试`
- `docs(config): 更新配置改造计划`

### 11.3 评审重点

代码评审优先关注：

1. 是否破坏边界
2. 是否引入新的兼容债务
3. 是否存在敏感信息风险
4. 是否缺少测试
5. 是否同步更新文档

## 12. 兼容与废弃规范

### 12.1 兼容原则

兼容必须有边界，不接受无限期保留的“临时兼容”。

要求：

- 明确兼容对象
- 明确兼容方式
- 明确废弃计划

### 12.2 废弃流程

建议流程：

1. 文档标记废弃
2. 保持读兼容
3. 收口写路径
4. 提供迁移说明
5. 在约定阶段移除

## 13. 安全规范

### 13.1 敏感信息

禁止：

- 硬编码密钥、密码、Token
- 在日志打印完整凭据
- 在文档中写入真实凭据

要求：

- 使用占位符
- 从安全存储获取真实值
- 通过脱敏方式展示必要信息

### 13.2 第三方依赖

新增依赖时必须说明：

- 解决什么问题
- 是否已有仓库内方案可复用
- 安全与维护风险是什么

## 14. 执行原则

本规范不是一次性文件，而是后续改造和开发的长期基线。

执行原则如下：

1. 新功能默认遵守本规范
2. 存量代码在修改时逐步向本规范收敛
3. 若规范与现实冲突，优先更新规范，再更新实现

## 15. 当前优先落地的规范项

建议先执行以下几条：

1. 新增 Markdown 文档统一使用中文命名
2. 停止继续向超大文件叠加职责
3. 配置模型、DTO、运行时对象开始分层
4. 聊天 SSE 协议单独建模
5. 敏感字段统一脱敏
6. 修 bug 强制补测试

以上六条应作为后续提交的最低执行标准。
